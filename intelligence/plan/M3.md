# M3: Cache Contract System

Technical specification for Milestone 3. Introduce `CacheContract` as a standardized per-phase cache interface. All phases use sha256-based contracts. CLI commands `cache --check` and `cache --target`.

---

## Current State (AS-IS)

### Per-phase ad-hoc skip logic

Each phase implements its own cache/skip logic internally:

| Phase | Skip mechanism | Files involved |
|-------|---------------|----------------|
| file-detection | Never skips (always re-detects) | — |
| graph | Compares `hashes.json` (incremental diffing) | `.cache/hashes.json` |
| outlines | Per-file SHA256 hash comparison | `.cache/outline-hashes.json` |
| communities | SHA256 of `graph-nodes.json` against stored hash | `.cache/graph-nodes-hash.txt` |
| enrich (M1) | SHA256(graph.json) + SHA256(config subset) + output existence | `.cache/enrich-input-hash.txt`, `.cache/enrich-config-hash.txt` |
| index | Checks if `graph_search.db` exists + mtime comparison | — (mtime-based) |
| embeddings | Config hash + `node-texts.json` diff | `.cache/embeddings-config-hash.txt`, `.cache/node-texts.json` |
| html | Mtime comparison of inputs vs outputs | — (mtime-based) |
| report | Mtime comparison of inputs vs outputs | — (mtime-based) |

### Problems

1. **No unified interface** — each phase invents its own skip logic
2. **Mtime-based checks** in some phases (unreliable across git clone, CI, Docker layers)
3. **No CLI introspection** — can't ask "is phase X stale?" without running the build
4. **No sealing mechanism** — phases don't explicitly record "I completed successfully with this input"
5. **Force bypass is implicit** — `ctx.force` checked inconsistently across phases

## Target State (TO-BE)

### Architecture

```
CacheContract (interface)
    ├── check(ctx) → { fresh: boolean, reason?: string }
    ├── seal(ctx) → void
    └── invalidate(ctx) → void

PhaseWithContract (extends Phase)
    └── contract: CacheContract

Orchestrator integration:
    1. Before execute(): if !force → contract.check()
    2. If fresh → skip (return PhaseResult with skipped=true)
    3. If stale → execute()
    4. After execute(): contract.seal()
```

### CacheContract interface

```typescript
// src/pipeline/cache/contract.ts

export interface CacheCheckResult {
  /** true = outputs are fresh, skip execution */
  fresh: boolean;
  /** Human-readable reason (for logging) */
  reason: string;
}

export interface CacheContract {
  /** Phase ID this contract belongs to */
  readonly phaseId: string;

  /**
   * Check if the phase's outputs are fresh (valid cache hit).
   * Returns fresh=true only when ALL conditions are met:
   *   1. All expected output files/dirs exist
   *   2. Input hash matches sealed hash
   *   3. Config hash matches sealed hash (if applicable)
   */
  check(ctx: CacheContext): CacheCheckResult;

  /**
   * Seal the cache after successful execution.
   * Writes hash files that future check() calls compare against.
   */
  seal(ctx: CacheContext): void;

  /**
   * Explicitly invalidate the cache (delete hash files).
   * Used by --force or when upstream changes are detected.
   */
  invalidate(ctx: CacheContext): void;
}
```

### CacheContext

```typescript
// src/pipeline/cache/context.ts

export interface CacheContext {
  /** Absolute path to output directory (reponova-out/) */
  outputDir: string;
  /** Absolute path to .cache/ directory */
  cacheDir: string;
  /** Full resolved config */
  config: Config;
}
```

## Per-Phase Contracts

### file-detection

```typescript
const fileDetectionContract: CacheContract = {
  phaseId: "file-detection",
  check: () => ({ fresh: false, reason: "always runs" }),
  seal: () => {},          // no-op
  invalidate: () => {},    // no-op
};
```

Rationale: file detection is cheap and must always re-scan for changed/added/removed files. It's the root of the DAG — everything flows from it.

## Integration into Phase Execution

### Option A: Orchestrator-driven (CHOSEN)

The orchestrator handles contract check/seal. Phases no longer contain any skip logic.

**File: `src/pipeline/engine/orchestrator.ts`** — `executePhase()`:

```typescript
async function executePhase(phase: Phase, ctx: PhaseContext): Promise<PhaseResult> {
  const contract = phase.contract;

  // Check cache (unless forced)
  if (contract && !ctx.force) {
    const cacheCtx: CacheContext = {
      outputDir: ctx.outputDir,
      cacheDir: join(ctx.outputDir, ".cache"),
      config: ctx.config,
    };
    const checkResult = contract.check(cacheCtx);
    if (checkResult.fresh) {
      log.info(`  [${phase.id}] Skipped: ${checkResult.reason}`);
      return { processed: 0, skipped: true, skipReason: checkResult.reason };
    }
  }

  // Execute
  const result = await phase.execute(ctx);

  // Seal cache after successful execution (not skipped, not error)
  if (contract && !result.skipped) {
    const cacheCtx: CacheContext = {
      outputDir: ctx.outputDir,
      cacheDir: join(ctx.outputDir, ".cache"),
      config: ctx.config,
    };
    contract.seal(cacheCtx);
  }

  return result;
}
```

### Phase interface update

**File: `src/pipeline/engine/phase.ts`**

```diff
+import type { CacheContract } from "../cache/contract.js";

 export interface Phase {
   readonly id: string;
   readonly label: string;
   readonly dependencies: string[];
+  /** Cache contract. If undefined, phase always runs. */
+  readonly contract?: CacheContract;
   execute(ctx: PhaseContext): Promise<PhaseResult>;
 }
```

### Migration from internal skip logic

Each phase currently has its own `shouldRun()`/skip checks inside `execute()`. These get REMOVED and replaced by the contract. The phases' `execute()` functions become pure "do the work" — they assume they should always run when called.

**Exception**: Fine-grained internal incrementality (per-file hashing in graph/outlines/embeddings) remains INSIDE the phase. The contract is only the coarse "should we enter this phase at all?" gate.

```
Contract.check() → "should we enter this phase?"        [COARSE]
Phase.execute()  → "what subset of work to do?"          [FINE-GRAINED, INTERNAL]
Contract.seal()  → "record that we completed"            [COARSE]
```

## CLI Commands

### `reponova cache`

New top-level command:

**File: `src/cli/cache.ts`** (NEW)

```typescript
import type { CommandModule } from "yargs";

export const cacheCommand: CommandModule = {
  command: "cache",
  describe: "Inspect and manage phase cache",
  builder: (yargs) =>
    yargs
      .option("check", {
        type: "string",
        describe: "Check if a phase's cache is fresh (exit 0 = fresh, exit 1 = stale)",
      })
      .option("seal", {
        type: "string",
        describe: "Manually seal a phase's cache (marks outputs as current)",
      })
      .option("invalidate", {
        type: "string",
        describe: "Invalidate a phase's cache (force re-run on next build)",
      })
      .option("status", {
        type: "boolean",
        describe: "Show cache freshness status for all phases",
        default: false,
      })
      .option("config", {
        type: "string",
        describe: "Path to reponova.yml",
      })
      .check((argv) => {
        const ops = [argv.check, argv.seal, argv.invalidate, argv.status].filter(Boolean);
        if (ops.length === 0) {
          throw new Error("Specify one of: --check, --seal, --invalidate, or --status");
        }
        if (ops.length > 1) {
          throw new Error("Only one operation at a time: --check, --seal, --invalidate, or --status");
        }
        return true;
      }),
  handler: async (argv) => { /* ... */ },
};
```

### `reponova cache --check <phase>`

- Load config → resolve output dir
- Instantiate the phase's contract
- Call `contract.check(cacheCtx)`
- If `fresh` → print "Phase <phase> is fresh: <reason>" → exit 0
- If stale → print "Phase <phase> is stale: <reason>" → exit 1

**Use case**: CI scripts can check if a phase needs re-running:
```bash
if ! reponova cache --check embeddings; then
  reponova build --start-after enrich
fi
```

### `reponova cache --seal <phase>`

- Load config → resolve output dir
- Validate that phase's expected outputs exist on disk
- Call `contract.seal(cacheCtx)`
- Print "Phase <phase> sealed." → exit 0

**Use case**: After manual intervention (e.g., editing graph-enriched.json by hand), seal the cache to prevent the phase from re-running.

### `reponova cache --invalidate <phase>`

- Load config → resolve output dir
- Call `contract.invalidate(cacheCtx)`
- Print "Phase <phase> invalidated." → exit 0

**Use case**: Force a specific phase to re-run on next build without `--force` (which forces ALL phases).

### `reponova cache --status`

- Load config → resolve output dir
- For each phase: call `contract.check(cacheCtx)`
- Print table:

```
Phase            Status    Reason
─────────────    ──────    ──────────────────────────
file-detection   stale     always runs
graph            fresh     inputs unchanged
outlines         fresh     inputs unchanged
communities      fresh     inputs unchanged
enrich           stale     config changed (threshold)
index            stale     input hash mismatch
embeddings       stale     input hash mismatch
html             stale     input hash mismatch
report           stale     input hash mismatch
```

For each phase:
1. Remove the internal `shouldRun()` / cache-check logic
2. Remove the early-return paths that produce `skipped: true`
3. Keep fine-grained incrementality (hashes.json, node-texts.json, outline-hashes.json)
4. The phase's `execute()` now assumes it WILL run when called

**Exceptions**:
- `file-detection`: no changes (already always runs)
- `graph`, `outlines`, `embeddings`: keep internal per-file diffing (fine-grained layer)

### `reponova cache`

Inspect and manage the per-phase build cache.

\```bash
reponova cache --check <phase>       # Check if phase cache is fresh (exit 0) or stale (exit 1)
reponova cache --seal <phase>        # Mark phase outputs as current (manual seal)
reponova cache --invalidate <phase>  # Force phase to re-run on next build
reponova cache --status              # Show cache freshness for all phases
\```

| Option | Description |
|--------|-------------|
| `--check <phase>` | Exit 0 if fresh, exit 1 if stale. Useful in CI scripts. |
| `--seal <phase>` | Manually seal after out-of-band modifications. Requires outputs present. |
| `--invalidate <phase>` | Delete cache hashes, forcing re-run. More targeted than `--force`. |
| `--status` | Print freshness table for all 8 phases. |
| `--config` | Path to `reponova.yml` (optional, auto-detected). |

**Example CI usage:**

\```bash
# Only rebuild embeddings if needed
if ! reponova cache --check embeddings; then
  reponova build --start-after enrich
fi
\```

## Design Decisions

### Why orchestrator-driven (not phase-driven)?

1. **Single responsibility**: Phases focus on "what to do", contracts focus on "when to do it"
2. **Consistency**: Can't forget to check cache — orchestrator always does it
3. **CLI introspection**: `cache --check` doesn't need to instantiate phases, just contracts
4. **Testing**: Contracts testable in isolation without running phase logic

### Why not mtime?

- Git checkout resets mtimes
- Docker layers have artificial mtimes
- CI machines may have clock skew
- SHA256 content hashing is deterministic and portable

### Why coarse + fine-grained layers?

- **Coarse** (contract): "Has ANYTHING changed since last run?" — cheap hash file comparison
- **Fine-grained** (internal): "WHICH items changed?" — per-file/per-node diffing within the phase

If coarse says fresh → skip entirely (zero I/O beyond reading 1-2 hash files).
If coarse says stale → enter phase, let internal logic minimize actual work.

### Hash file naming convention

Pattern: `.cache/<phase>-<qualifier>-hash.txt`

- `<phase>`: phase ID (e.g., `enrich`, `graph`, `html`)
- `<qualifier>`: `input` (hash of input file), `config` (hash of config subset)

Examples:
- `.cache/enrich-input-hash.txt`
- `.cache/enrich-config-hash.txt`
- `.cache/graph-input-hash.txt`
- `.cache/index-input-hash.txt`
